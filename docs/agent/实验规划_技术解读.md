# ChatPathway2 实验规划（技术解读与执行细节）

本文档固定科学问题、对照关系、执行顺序和结果闸门。代码层的唯一机器可读矩阵是
`experiments/matrix.json`；服务器实时状态、数据 hash 与当次结果记录在
[`STATUS.md`](STATUS.md)。数据与模型契约见
[《项目 SPEC》](../项目SPEC.md)。

## 1. 当前核心问题

第一轮不是“尝试所有 latent dynamics”，而是严格回答：

> 在完全共享的 Qwen3-8B 第一阶段 SFT 和重建 AE 之上，第二阶段加入 graph-layer
> HNN 或 `(J-rI)∇H+F(t)` 辅助目标，是否比“仅多训一次 SFT”更好？

主方法是 forced-damped HNN，纯 HNN 是结构对照，第二阶段 SFT-only 是计算量/附加微调
对照，第一阶段 SFT 是直接生成基线。没有这四条臂的同 seed 对比，就不能把任何改善
归因给 Hamiltonian 结构。

## 2. 固定不变的公共设置

| 项目 | 第一轮固定值/政策 |
| --- | --- |
| base model | 固定 revision 的 Qwen3-8B |
| 训练数据 | 同一 dataset revision、split manifest 与选样 manifest |
| 文本预算 | prompt+answer `max_length=8192` |
| 每进程 batch | 1，不为某一臂单独降配 |
| semantic unit | ordered graph layer；同层 event set 先 pooling |
| dynamics horizon | 最多 128 个 layer advance，`dt=1/128` |
| 推理 | validation-selected LoRA 的 direct greedy generation |
| 推理 batch | 每进程 1；四卡使用互斥数据 shard |
| validation | 按 `pathway_family_id` 整组留出，同 seed 全阶段复用 |
| checkpoint | 只报告 validation-selected `checkpoint_best` |
| 重复 | 正式 seeds: `20260711, 20260712, 20260713` |
| phenotype | 关闭；缺失不作为阴性 |

数据 revision 一旦改变，旧 checkpoint 只能作为另一数据版本的 baseline，不得与新数据臂混在同一
受控矩阵中。特别是 uncapped、family cap 32 和 family cap 64 是三个不同数据选择版本，
必须分别记录。

## 3. 第一轮五个可执行 row

| ID | 训练 | 推理 | 它真正测试什么 |
| --- | --- | --- | --- |
| `base000_shared_sft_reconae` | 四卡 DDP 第一阶段 SFT，然后训共享 4096-128-4096 AE | 只验证 artifact | 为同 seed 所有 stage-2 臂提供完全相同的起点 |
| `exp000_sft_only_direct` | 直接复用 stage-1 SFT | direct greedy | 第一阶段 SFT 基线，不是计算匹配对照 |
| `exp003_stage2_sft_only_direct` | 与 dynamics 臂相同的 stage-2 loop，但 dynamics loss 权重和 LR 为 0 | direct greedy | 排除“只是多训了一阶段 SFT” |
| `exp001_hnn_reconae_joint_direct` | stage-2 LoRA + `dz/dt=J∇H` 联合优化 | direct greedy，不加载 HNN | 保守 Hamiltonian semantic-layer 正则化的增量 |
| `exp002_forced_damped_hnn_reconae_joint_direct` | stage-2 LoRA + `(J-rI)∇H+F(t)` 联合优化 | direct greedy，不加载 dynamics | 主方法：耗散与 time-only forcing 是否超过 HNN 与 SFT-only 对照 |

每个 seed 的依赖关系是：

```text
base000: shared SFT -> shared AE
       |-> exp000: stage-1 direct
       |-> exp003: compute-matched stage-2 SFT only -> direct
       |-> exp001: stage-2 SFT + HNN             -> direct
       `-> exp002: stage-2 SFT + FDHNN           -> direct
```

`exp003`、`exp001`、`exp002` 必须在同 seed 下使用相同的 row/prefix sampling 规则、epoch 数、
LoRA optimizer schedule、validation family 和 SFT/AE digest。唯一主要差别是动力学目标。

### 3.1 数据冻结后的 D3：dynamics 预训练再联合

E011/E021 使用同一份 validation-selected SFT、冻结 AE 和同一 dataset revision，执行：

1. 冻结 LoRA 和 AE，仅训练 HNN/FDHNN。至少 1 轮、最多 3 轮；每轮验证 rollout/state/
   velocity 与结构正则，保存 validation dynamics composite 最优者。只有无 NaN/Inf、覆盖率
   达标且 validation 指标达到预先固定的稳定判据时，结果才标记为 `stability_passed`；达到
   3 轮仍未满足时必须如实写 `max_epochs_without_stability`。
2. 从最佳 dynamics checkpoint 开始联合训练。LoRA 学习率在实现阶段用 validation 从
   `{5e-6, 1e-5}` 选择一次，随后对全部正式 seed 固定；dynamics 学习率为 `2e-4`。AE 与
   gold target branch 始终冻结。
3. 使用两个优化器或等价的显式梯度路由。dynamics 参数从联合阶段第一步起继续接收完整
   `L_dyn`；只有 `L_dyn -> LoRA` 的系数在联合训练前 10% optimizer steps 从 0 线性升到
   目标值。不能把总动力学损失整体乘 0，否则 warmup 期间 dynamics 也会停止学习。
4. 冻结第一阶段 SFT adapter 作为 reference teacher，在 assistant supervised tokens 上计算
   `KL(p_stage1 || p_joint)`；prompt/padding token 不计入 KL。KL 权重由 validation 预先固定，
   不得观察 test 后修改。

联合阶段需要同时保存四个恢复点：

- `checkpoint_best_sft`：validation SFT CE 最低，只作诊断；
- `checkpoint_best_dynamics`：validation dynamics composite 最低，只作诊断；
- `checkpoint_best_composite`：按最终固定损失权重计算的 validation composite 最低，是唯一
  默认主结果与 test inference 输入；
- `checkpoint_last`：只用于中断恢复，不参与结果选择。

每 100 个 optimizer steps 在同一批次上分别计算 LoRA 参数的 `g_sft` 与 `g_dyn`，以 FP32
累计并记录 `cos(g_sft,g_dyn)`、两者范数及负余弦比例。DDP 下只归约点积和范数，不保存完整
梯度向量。梯度夹角是冲突诊断，不直接作为模型优劣指标，也不允许事后据此挑 test checkpoint。

该方法在结果表中统一称为“dynamics 预训练后联合训练（D3）”。只有
`stability_passed=true` 时，文字说明才可写“HNN 在固定 SFT+AE 隐空间达到稳定后联合”；
否则必须报告未达到稳定判据。当前机器矩阵尚未实现 D3，本节只是冻结下一轮方法，不改变
第一轮可执行的 D4 配置。

## 4. 执行顺序与硬闸门

### 4.1 数据物化

开始任何新数据版本的正式训练前，必须完成：

1. 固定 input inventory、dataset/split/selection manifest 和 SHA-256；
2. 验证 identity、schema、family/cluster overlap 和 leading-zero 字符串读取；
3. 使用与训练相同的 tokenizer/max length 审计 layer/event 保留率；
4. 确保每个 assistant target 是完整闭合 JSON，不在 token 中间截断；
5. 确保 validation 是固定的每 record 一个 prefix，不随 epoch 漂移；
6. 记录同一 dataset revision 下的 short/middle/long horizon 分布。

当前 v2 仍存在 token 截断可能产生不闭合 assistant JSON 的阻塞。对新 v3 数据不得带着该问题
启动正式矩阵。

### 4.2 运行前审计与小规模贯通

从仓库根目录执行：

```bash
python -m experiments.validate_matrix
python -m experiments.run_experiment audit --quiet
python -m experiments.run_experiment consistency --phase both --quiet
```

然后用一个明确标记为 disposable 的 seed 顺序贯通 SFT -> AE -> HNN -> FDHNN -> checkpoint
reload -> direct generation。小规模运行只是“工程贯通测试”，不是正式结果；不再使用意义不明的
英文简称来向用户描述它。

贯通闸门包括：

- 前向、反向和 tail gradient accumulation 都能完成；
- 没有 NaN/Inf，有效样本不应全部变成 zero semantic layers；
- AE、LoRA 和 dynamics checkpoint 能重新加载；
- 输出 JSON/schema validity、finish reason、token 数和 source identity 可审计；
- 四卡上的 loss/reduction 与单卡参考无量级异常；
- 中断后重启不覆盖其他 seed/revision 产物。

### 4.3 正式三 seed 矩阵

对每个 seed 先只生成一份 shared SFT 和 AE，再运行四条比较臂。不同 seed 的 artifact
写入独立目录，不相互覆盖。每个 trainer 都必须设置：

- Python/NumPy/PyTorch/CUDA/DDP 可控的随机种子；
- family-grouped validation；
- 基于 validation 主指标的 early stopping；
- `checkpoint_best` 与 `best_checkpoint.json`；
- config、Git commit、base revision、data/SFT/AE/dynamics digest；
- 逐 epoch/step 机器可读 metrics JSONL 与清晰文本 log；
- supervised tokens、throughput、截断、semantic coverage、训练/验证耗时。

当前 checkpoint 保存不包含可证明“精确恢复同一轨迹”的完整 optimizer/scheduler/RNG 状态，
所以 warm-start 不得写成 exact resume。

### 4.4 标准执行入口

数据准备、候选 coverage revision、base model 下载和严格 asset check 均通过统一入口：

```bash
export CHATPATHWAY_PROFILE=cfff

python -m experiments.run_experiment prepare-data --overwrite
python -m experiments.run_experiment prepare-coverage \
  --max-records-per-family 64 --seed 20260711 --overwrite
python -m experiments.run_experiment download-model
python -m experiments.run_experiment check-assets \
  --phase train --ids base000_shared_sft_reconae --profile cfff \
  --create-output-dirs --strict
```

`prepare-coverage` 只生成候选 dataset revision 和明确 validation 文件，不会也不得静默替换
矩阵默认输入。晋升 cap64 时要同时更新 dataset manifest、matrix/profile 和 checkpoint namespace。

单 seed 的显式顺序：

```bash
SEED=20260711
python -m experiments.run_experiment train base000_shared_sft_reconae -- --seed "$SEED"
python -m experiments.run_experiment train exp003_stage2_sft_only_direct -- --seed "$SEED"
python -m experiments.run_experiment train exp001_hnn_reconae_joint_direct -- --seed "$SEED"
python -m experiments.run_experiment train exp002_forced_damped_hnn_reconae_joint_direct -- --seed "$SEED"

python -m experiments.run_experiment infer exp000_sft_only_direct -- --seed "$SEED"
python -m experiments.run_experiment infer exp003_stage2_sft_only_direct -- --seed "$SEED"
python -m experiments.run_experiment infer exp001_hnn_reconae_joint_direct -- --seed "$SEED"
python -m experiments.run_experiment infer exp002_forced_damped_hnn_reconae_joint_direct -- --seed "$SEED"
```

CFFF 四卡依赖感知调度：

```bash
python -m experiments.run_cfff_matrix \
  --seeds 20260711,20260712,20260713 \
  --gpus 0,1,2,3
```

先使用 `--dry-run` 检查完整命令图。只在 trainer 的 atomic `run_complete.json` 与必需
checkpoint 同时存在时才跳过已完成任务；非空但不完整的输出目录必须 fail closed，
不自动当作成功或覆盖。

## 5. 四张 A100 怎样用

当前调度可以让节点长时间保持四卡繁忙，但不能说每个程序都已做四卡分布式训练：

- 第一阶段 SFT 使用四卡 DDP，四卡参与同一个优化任务；
- AE 和当前 HNN/FDHNN stage-2 臂仍主要是单 GPU 任务；
- 在共享 SFT 完成后，调度器将互相独立的 AE/stage-2/inference 工作分配到不同 GPU，
  以任务并行占满节点；
- 直接推理在四卡上分数据 shard，每进程 batch size 仍为 1，最后严格合并。

因此“能否占满四张 A100”的答案是：节点级调度可以；但当前只有 SFT 是同一 job 的四卡 DDP，
不得把单卡臂的并行调度宣称为四卡 dynamics 分布式算法。

长时运行应使用 tmux 或等价会话保持器；`nohup` 也能防止 SSH 断开导致进程退出，
但仍要有 PID、stdout/stderr 路径、状态检查和失败重启记录。

## 6. 评价与结果归因

### 6.1 训练/生成主指标

- grouped validation CE/complete objective 用于 checkpoint 选择；
- 贪心生成的 JSON validity、schema validity、finish reason 和长度分布；
- exact/normalized event 与 layer-set 结构指标；
- strict family-disjoint core 与 organism-transfer 分开报告；
- 三 seed 汇总均值、离散度和每 seed 原始值，不只报最好一次。

### 6.2 表示与动力学指标

- Task 0：AE reconstruction MSE/cosine，HNN/FDHNN 在固定 horizon 的 rollout error；
- Task 2：同一 base/adapter/AE representation 下 prediction/gold trajectory 的 DTW PCTE；
- force/damping 大小、energy-balance residual 和 vector-field residual 只是诊断；
- `H` 不是生化自由能，energy monotonicity 不是因果方向标签；
- Task 3 中的 HNN 分数只有在 validation 拟合并冻结 calibration 后才能与 LLM score 组合。

### 6.3 最低可报告门槛

- `exp001/exp002` 同时对比 `exp000` 和 `exp003`；
- 每条臂共享前置 checkpoint digest 经过验证；
- 只使用 `checkpoint_best`，不把任意 `checkpoint_epoch_N` 叫做 best；
- split/identity/token retention/coverage 审计全部通过；
- 失败或排除的样本有数量、identity 和原因；
- 数据、代码、配置、base model 和 checkpoint 均有不变标识；
- 结论不超过数据契约：v2 只支持 graph-layer 或 parser-audited 说法。

## 7. 第一轮后保留的实验轴

以下方向是明确延后，不是删除。它们必须在当前矩阵和对应数据契约完成后单独建矩阵。

### 7.1 粒度轴

1. **graph-layer-by-graph-layer**：实现可验证的 JSON layer-boundary controller，每完成一层调用一次现有 dynamics；
2. **atomic event sub-step**：等 v3 canonical event 到位后训练独立 event-resolution trajectory，不从句子顺序伪造时间；
3. **token-by-token**：单独训练 token-resolution vector field，禁止将 graph-layer checkpoint 每 token 调用；
4. **multiscale hybrid**：前三者分别验证后组合，并做组件消融。

### 7.2 AE 轴

- 共享 reconstruction AE（当前基准）；
- 对 semantic layer/event set 更明确的表示学习；
- 冻结 teacher target 与当前 stop-gradient moving target 对照；
- latent dimension、reconstruction/geometry loss 和正交 Poisson frame 稳定性消融。

AE 轴不能与 dynamics 轴随意一次改多项，否则无法知道改善来自更好重建还是新向量场。

### 7.3 dynamics 轴

- pure HNN 与 forced-damped HNN（当前）；
- Neural ODE、gradient flow、GENERIC、Koopman、SINDy 等已有原型，但暂不插入第一轮归因；
- PHNN 只有在数据定义了独立观测 port/control `u`、单位、时序和监督后才能实现。

上游 pathway 现在只用于构造初始状态 `z0`，它不是独立外部 port。因此不能将 prompt embedding
换个名字叫 `u` 就声称完成 PHNN。

### 7.4 推理轴

- direct greedy（当前基准）；
- 多候选生成 + validation-calibrated rerank；
- 只在语义边界一致时的 rollout residual/injection；
- 同等 decoding token、candidate 数和 wall-clock budget 下的受控比较。

### 7.5 prompt 轴

- metadata-rich：给 pathway title/ID/class + observed events；
- prefix-only：只给 observed structured events；
- schema skeleton vs schema-constrained decoding；
- 完整 layer/horizon 物化策略。

metadata-rich vs prefix-only 是必做消融，用来分离“从预训练记忆检索 KEGG 通路”与“根据 observed
events 推理下游”。

## 8. 当前不得启动的结果声明

- 没有 independent `u` 时的 PHNN 结果；
- 没有真实干预和校准 phenotype scorer 时的 Task 4 accuracy；
- 将 missing phenotype 当 negative 的任何指标；
- 未经专家验证/provenance 的 Task 3 direction/shuffle/unrelated negative；
- 将 same-layer sentence order 当作生物时间的 sub-step 结果；
- 将 graph-layer checkpoint 每 token 推进的 inference-time HNN 结果；
- 不同 dataset revision 或不同 validation split 之间无限定的 validation loss 横向比较；
- 无 manifest 的旧 checkpoint 对其精确训练算法的追溯性声明。

第一轮的完成标志不是“五个脚本都跑过”，而是三个 seed 在同一数据契约下完成严格共享前置、
对照、validation 选择、不变 artifact 记录与可复核下游评估。
